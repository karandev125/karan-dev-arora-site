This is the first public Thought on the site, but it is also a small test object. A personal site is easier to maintain when its first post proves the format instead of merely announcing that the format exists. So this page includes the ordinary things I expect to use later: paragraphs, headings, code, a footnote, a quote, an internal link back to the [Thoughts index](/thoughts), and an external link to [nav.al](https://nav.al), whose plainness is one of the references for the visual direction here.

The point is not to make the site feel empty or ceremonial. The point is to make it feel ready. If I come back months from now with a note about Go, databases, or the strange patience required to learn hard things, the page should already know how to hold it. It should not ask me to decorate the thought before I can publish it.

## A Quiet Format

The site is intentionally narrow. The measure is close to what I like for essays: not so wide that the eye drifts, not so narrow that every sentence breaks into a staircase. Most of the design lives in small decisions: the cream background, the ink-blue links, the serif body, the mono metadata, and the absence of boxes around everything.

That last part matters. Cards are useful when the thing being shown is an object with multiple independent attributes. Essays are not objects in that sense. They are closer to a line of attention. The page should make the next sentence feel available, then the next one, then the next one after that.

Inline code should sit quietly inside a sentence, like `const draft = false`, without becoming a badge. Code blocks need more structure because they have their own local context. A filename helps the reader understand whether the snippet is a utility, a component, or a configuration file.

```ts filename="src/lib/reading-time.ts"
const WORDS_PER_MINUTE = 220;

export function readingTime(wordCount: number): string {
  const minutes = Math.max(1, Math.ceil(wordCount / WORDS_PER_MINUTE));
  return `${minutes} min read`;
}
```

The code block above is deliberately plain. It should be legible, copyable, and warm enough to belong to the page. It should not look like a glowing terminal dropped into the middle of an essay. Syntax highlighting helps only when it reduces effort; once it starts performing for attention, it is working against the writing.

> A useful personal site should feel like a desk that has been cleared for thinking, not a showroom for proving that the desk exists.

That is also why the footer is small, the navigation is textual, and the homepage does not have a hero image. A personal website can be memorable without shouting. Sometimes the strongest signal is restraint: a name, a few links, a sentence with a point of view, and enough silence around the words for them to breathe.

## Metadata Is Part of the Voice

The line under the title is mono because it is not part of the essay's music. It is a small instrument panel: date, reading time, and later an updated date if the piece changes meaningfully. The date answers "when was this written?" and the reading time answers "how much room should I make for it?" Both should be visible without becoming important.

Reading time is only a rough estimate, but rough estimates are useful. If a post is five minutes long, I want that promise to be honest enough. If it grows into a longer essay, the format should not need a redesign. It should simply make room for a table of contents and keep the body stable.

### Headings Should Be Linkable

Headings are addresses. If a section is worth naming, it is probably worth linking to. The little `#` that appears beside a heading should feel like a tool left nearby, not a decoration. It appears when needed, stays out of the way otherwise, and copies the anchored URL for anyone who wants to point to a specific section.

Footnotes work the same way. On a narrow screen they belong at the bottom, because the reader has no margin to spare. On a wider screen, the note can move beside the sentence that called it.[^sidenote] That is the more natural reading pattern: glance aside, then return.

[^sidenote]: This is the first real test of the sidenote treatment. On desktop it should sit in the right margin near the reference; on smaller screens it should remain a normal numbered footnote.

There is a small danger in building a system like this: it can become a project about the system instead of the writing. That is not the goal. The goal is a structure that disappears quickly. Once the details are reliable, I should be able to open a file, write the thing, and trust the site to make it readable.

The homepage says I am interested in computer science, AI, backend engineering, and Go. That is true, but the site should be broad enough to hold adjacent questions too: why some abstractions feel tasteful, why simple tools last, why learning often feels slow right before it becomes useful. Those questions do not need a loud interface. They need a place that rewards returning.

So: hello. This post is partly a greeting, partly a fixture test, and partly a reminder to keep the site honest. If the design is doing its job, the next post can be about something else entirely.
